The complexity of a programming language

Published on 1970-1-1

The other day I was at the Kotlin workshop in London ran by @hhariri, and the first thing that was covered was:

We're making Kotlin because we want the constructs from languages like Clojure or Scala but done so ordinary developers can use them

I paraphrase of course and it's a noble goal, but after about an hour of being shown a million different programming language constructs in Kotlin I tweeted:

I'm sure it's not, but so far Kotlin just looks like somebody vomited a load of language ideas over the java runtime

— 'Hurricane Rob' (@RobAshton) May 2, 2013

What makes a programming language complicated?

For me? How much do I need to know in order to use it?

I don't tend to stick to one particular programming language religiously, I will happily work in anything a client wants me to and in order to keep this flexibility I try to use the same tooling for all of them. (vim + terminals + a browser).

The barrier to getting stuff done in any of these programming languages can be directly linked to how many constructs are available in that language for developers to abuse.

I've described Ruby (and CoffeeScript) in the past as being languages where

Ruby is the language where you have 16 different ways to write a for loop

Now, I'm about to start working on a project that uses Ruby extensively so I'm not a hater, but this kind of thing irks me when I'm picking up something new.
